# Implementation Plan: Product Catalog Management

**Branch**: `002-products` | **Date**: 2026-06-27 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/002-products/spec.md`

---

## Summary

Replace the Products placeholder page with a fully functional product catalog: a
searchable, filterable table with per-row Add / Edit / Delete-or-Deactivate actions
and an active/inactive lifecycle that is consistently enforced across the Sales and
Purchases product pickers built in later phases. Three new source files are introduced
(`products_db.py`, `ui/product_dialog.py`, `ui/products_page.py`); the placeholder
`products_page.py` is completely overwritten.

All technology decisions are governed by the approved `BLUEPRINT.md`. No research
unknowns exist — see `research.md` for the rationale record.

---

## Technical Context

**Language/Version**: Python 3.x (same as Phase 1)

**Primary Dependencies**: PySide6 (UI), `sqlite3` stdlib (data access) — no new
dependencies

**Storage**: SQLite `data/store.db` — `Products` table created in Phase 1

**Testing**: Manual end-to-end per `quickstart.md` Phase 2 checkpoint list

**Target Platform**: Windows 10 single-user desktop

**Project Type**: Desktop GUI application (incremental feature addition to Phase 1
shell)

**Performance Goals**: Add/Edit/Delete complete in < 2 s; search filters in < 1 s
(SC-001–SC-004)

**Constraints**: No ORM; all SQL parameterized; layering rules from Constitution
Principle I; single shared `get_active_products()` function (Constitution Principle IV)

**Scale/Scope**: Single Admin, catalog of up to ~thousands of products on local SQLite

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-checked after Phase 1 design.*

| Principle | Gate | Status |
|---|---|---|
| I. Layered Architecture | `products_db.py` (data access) MUST NOT contain business rules. `ui/products_page.py` and `ui/product_dialog.py` MUST NOT import `sqlite3`. `ui/products_page.py` MAY import `products_db.py` directly (no logic layer needed for CRUD without calculated fields). | ✅ All three files assigned to correct layers; no cross-layer violations. |
| II. Fixed Technology Stack | Python + PySide6 + SQLite only; all SQL parameterized; short-lived connections. | ✅ No new dependencies. |
| III. Single-User Scope | No roles or permissions introduced. | ✅ N/A |
| IV. Financial Data Integrity | `is_active` MUST be a single global rule. Exactly one shared `get_active_products()` function must exist in `products_db.py`, reused identically by Sales (Phase 4) and Purchases (Phase 3). Hard-delete is only allowed if the product has zero rows in both `SaleItems` and `PurchaseItems`. | ✅ `is_product_referenced()` guards the delete path; `get_active_products()` is the sole picker function. |
| V. Fail Safely, Never Silently | Duplicate-SKU `IntegrityError`, failed delete, and all validation errors surfaced as `QMessageBox`; no raw traceback. | ✅ Required in both `product_dialog.py` and `products_page.py`. |
| Phase Completion Rule | All Phase 1 features must remain working; Phase 2 must be demoable end-to-end before Phase 3 starts. | ✅ Phase 2 checkpoint list in `quickstart.md` must pass. |

**Post-Design Re-check**: All principles satisfied. No complexity justification required.

### Source Code (new/changed files only)

```text
accounting_system/
│
├── products_db.py           # NEW — Data access layer: all Products SQL queries
│
└── ui/
    ├── product_dialog.py    # NEW — Add/Edit modal dialog
    └── products_page.py     # REPLACE placeholder — full Products page
```

**Structure Decision**: Same flat layout as Phase 1. No `logic/` file is needed for
this phase — product CRUD has no multi-step transactions or calculated fields that
require a separate logic layer. All business rule enforcement (validation, delete
guard, active-filter) lives in `products_db.py` (data access) or in `product_dialog.py`
(input validation before any DB call).

### File Build Order (strictly bottom-up)

| # | File | Layer | Purpose | Depends On |
|---|---|---|---|---|
| 1 | `products_db.py` | Data Access | All parameterized SQL for the Products table | `database.py` (Phase 1) |
| 2 | `ui/product_dialog.py` | UI | Add/Edit modal: fields, validation, save/cancel | `products_db.py` |
| 3 | `ui/products_page.py` | UI | Replaces placeholder: table, search, Show Inactive, row actions | `product_dialog.py`, `products_db.py` |

### File Dependency Diagram

```
ui/products_page.py
  ├─> ui/product_dialog.py ─> products_db.py ─> database.py
  └─> products_db.py
```

## Complexity Tracking

> No constitution violations requiring justification in this phase.
> Note: UI importing `products_db.py` directly (bypassing a logic layer) is permitted
> here because product CRUD involves no multi-table transactions, no calculated
> business fields, and no stock rules — all of which would warrant a logic layer.
> The delete guard (`is_product_referenced()`) is a pure query, not business logic.
