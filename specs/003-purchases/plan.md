# Implementation Plan: Purchase Invoice Management

**Branch**: `003-purchases` | **Date**: 2026-06-27 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/003-purchases/spec.md`

---

## Summary

Replace the Purchases placeholder page with a full purchase-invoice creation screen.
The Admin selects active products from a picker, enters quantities and unit prices,
builds an in-progress invoice in memory, then saves it as an atomic transaction that
simultaneously writes the invoice header, all line items, increments each product's
stock quantity, and updates each product's purchase price. Three new source files are
introduced (`purchases_db.py`, `logic/purchase_logic.py`, `ui/purchases_page.py`);
the placeholder is completely overwritten.

This phase also implements `void_purchase()` in the data access and logic layers
(used by Phase 5's Invoice Detail Dialog) while leaving it unexposed in the Phase 3 UI.

---

## Technical Context

**Language/Version**: Python 3.x (same as Phases 1–2)

**Primary Dependencies**: PySide6 (UI), `sqlite3` stdlib (data access) — no new
dependencies

**Storage**: SQLite `data/store.db` — `Purchases`, `PurchaseItems` tables created in
Phase 1; `Products` table modified (stock_quantity + purchase_price updates) on save

**Testing**: Manual end-to-end per `quickstart.md` Phase 3 checkpoint list

**Target Platform**: Windows 10 single-user desktop

**Project Type**: Desktop GUI application (incremental feature addition)

**Performance Goals**: Save Invoice (including all stock updates) completes in < 3 s
for up to 20 line items (SC-004)

**Constraints**: No ORM; all SQL parameterized; single atomic transaction for save;
logic layer mandatory (unlike Phase 2); in-progress invoice retained on navigation (FR-026)

**Scale/Scope**: Single Admin, catalog up to ~thousands of products, each invoice up
to ~20 lines in practice

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-checked after Phase 1 design.*

| Principle | Gate | Status |
|---|---|---|
| I. Layered Architecture | `purchases_db.py` (data access) MUST NOT contain business rules or PySide6 imports. `logic/purchase_logic.py` (logic) MUST NOT contain raw SQL or PySide6 imports. `ui/purchases_page.py` MUST NOT import `sqlite3` directly and MUST call only `logic/purchase_logic.py` and `products_db.get_active_products()`. | ✅ All three files assigned to correct layers; no cross-layer violations planned. |
| II. Fixed Technology Stack | Python + PySide6 + SQLite only; all SQL parameterized; single atomic transaction for save (header + lines + stock + price update). Short-lived connections throughout. | ✅ No new dependencies. Transaction wraps all steps of a save. |
| III. Single-User Scope | No roles, no concurrent users, no network. | ✅ N/A |
| IV. Financial Data Integrity | Purchases are never deleted — `void_purchase()` sets `status = 'voided'` and reverses stock. Void is blocked if stock already sold. One void per invoice. `get_active_products()` shared function from Phase 2 used unchanged for the picker. | ✅ Void logic implemented now; UI for void deferred to Phase 5 per plan. |
| V. Fail Safely, Never Silently | Zero-line save blocked before any DB call. Quantity > 0 and unit price >= 0 validated in logic layer. DB errors caught in UI, shown as QMessageBox, in-progress form preserved on screen. | ✅ All error paths handled in logic + UI layers. |
| Phase Completion Rule | All Phase 1 and Phase 2 features must remain working. Phase 3 must be demoable end-to-end (Checkpoint 20) before Phase 4 begins. | ✅ Phase 3 checkpoint list in quickstart.md must pass. |

**Post-Design Re-check**: All principles satisfied. No complexity justification required.

| # | File | Layer | Purpose | Depends On |
|---|---|---|---|---|
| 1 | `purchases_db.py` | Data Access | All parameterized SQL for Purchases + PurchaseItems tables | `database.py` (Phase 1), `products_db.py` (Phase 2) |

| 3 | `ui/purchases_page.py` | UI | Replaces placeholder: picker, line-items table, Clear Invoice, Save Invoice, wait cursor | `logic/purchase_logic.py`, `products_db.py` |
