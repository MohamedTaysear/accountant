# Implementation Plan: Foundation, Shell & Auth

**Branch**: `001-foundation-shell-auth` | **Date**: 2026-06-27 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/001-foundation-shell-auth/spec.md`

---

## Summary

Establish the entire technical foundation and the full application shell so that every
later phase only ever adds page content — never new plumbing. This phase delivers a
fully launchable, securable, and navigable application: database initialisation and
admin seeding, login/logout, the main window shell with placeholder pages, the Change
Password dialog, and the Backup Now action.

All decisions are pre-decided by the approved `BLUEPRINT.md` and `IMPLEMENTATION_PLAN.md`.
No research was required — see `research.md` for the rationale record.

---

## Technical Context

**Language/Version**: Python 3.x (project-standard; no version constraint beyond what
PySide6 requires)

**Primary Dependencies**: PySide6 (UI framework), standard library only (`hashlib`,
`hmac`, `secrets`, `shutil`, `sqlite3`, `sys`, `os`, `pathlib`, `datetime`)

**Storage**: SQLite via the standard `sqlite3` module — file `data/store.db` created
at runtime relative to the project root

**Testing**: Manual end-to-end testing per the Phase 1 checkpoint checklist in
`quickstart.md`; no automated test suite in scope for this phase

**Target Platform**: Windows 10 single-user desktop (single local machine, no network)

**Project Type**: Desktop GUI application

**Performance Goals**: Login response < 1 s; first-launch database creation < 3 s;
Backup Now < 2 s — all on local disk

**Constraints**: Single-user, offline-only, no ORM, no connection pooling, all SQL
parameterized; Main Window min 900×600; Login Window fixed 400×250

**Scale/Scope**: Single Admin account, single SQLite file, no concurrent access

---

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-checked after Phase 1 design.*

| Principle | Gate | Status |
|---|---|---|
| I. Layered Architecture | UI files MUST NOT import `sqlite3` or `database.py` directly (one documented exception: `main_window.py` → `database.backup_database()`). Logic files MUST NOT import PySide6 or contain raw SQL. | ✅ All 13 Phase 1 files assigned to the correct layer in the file table below. |
| II. Fixed Technology Stack | Python + PySide6 + SQLite only; stdlib for hashing/backup; no ORM; all SQL parameterized; short-lived connections only. | ✅ No third-party dependencies beyond PySide6 introduced. |
| III. Single-User Scope | One Admin row, no registration, no roles. | ✅ `seed_default_admin()` creates exactly one row, guarded by idempotency check. |
| IV. Financial Data Integrity | Not applicable in Phase 1 — no invoices yet. | ✅ N/A |
| V. Fail Safely, Never Silently | All critical actions (login, backup, change password) wrapped in top-level handler; friendly `QMessageBox` on any error; no raw traceback to user. | ✅ Required in all relevant files (see file table). |
| Phase Completion Rule | App MUST be fully runnable and manually testable end-to-end at phase end. | ✅ All 10 checkpoint items in `quickstart.md` must pass before Phase 2. |

**Post-Design Re-check**: All principles satisfied. No complexity justification required.

**Structure Decision**: Single flat project — no src/ indirection. All source files
live directly under `accounting_system/` with `ui/` and `logic/` as the only
subdirectories. This matches the Blueprint's file structure exactly.
